# ADR-0005: Pokemon Champions専用ツールとする

## ステータス

**承認**

| 日付       | ステータス | 備考 |
| ---------- | ---------- | ---- |
| 2026-03-30 | 提案       |      |
| 2026-03-30 | 承認       |      |

## コンテキスト

ADR-0004で能力ポイント制（Pokemon Champions仕様）の採用を決定し、影響欄にて「Champions専用化の判断については別ADRで決定する」と予告した。本ADRはその決定を行うものである。

現状の実装・設計は既にPokemon Champions専用になっている:

- **ドメイン層**: `StatPoints`（能力ポイント、各32上限・合計66上限）、`StatAlignment`（能力補正25種）を採用し、個体値・努力値・性格は廃止済み
- **計算ロジック**: `StatsCalculator`はChampions専用の計算式（HP: 種族値+75+能力ポイント）、`DamageCalculator`はLv.50固定
- **要件定義**: `Claude要件定義.md`は「対象世代: Pokemon Champions」と明記済み
- **CLAUDE.md**: プロジェクト概要に「ポケモン対戦（Pokemon Champions）を補助するWebアプリケーション」と記載済み

しかし、この「Pokemon Champions専用」という方針はプロジェクト全体の正式な決定として文書化されていない。`ざっくり要求.md`にはまだ「対象世代: SV」の記載が残っており、プロジェクト名は「PokemonTools」と汎用的なままである。暗黙の方針が未文書化のままでは、レビュー基準・命名・モデル設計・テスト方針が揺れるリスクがある。

## 検討した選択肢

### 選択肢1: Pokemon Champions専用を正式決定（現状追認）

- 概要: 本ツールのスコープをPokemon Championsに限定し、全機能（マスターデータ、実数値計算、ダメージ計算、UI入力項目、インポート/エクスポート形式）をChampions仕様前提とする
- メリット:
  - 暗黙の方針を明文化し、設計判断の基準を統一できる
  - 不要な抽象化を排除し、ドメインモデルと計算ロジックを簡素化できる
  - スコープの明確化により開発の意思決定が迅速になる
- デメリット:
  - 過去作品のデータや計算ロジックの流用性が低下する
  - 将来別作品に対応する際は再設計コストが発生する

### 選択肢2: 将来の多世代対応を見据えた抽象化レイヤーの導入

- 概要: 世代ごとの差分を吸収する抽象化レイヤー（計算戦略インターフェース等）を導入し、将来的にChampions以外のタイトルにも対応可能にする
- メリット: 将来別作品が登場した際、同一コードベースで対応できる可能性がある
- デメリット:
  - 抽象化対象が不明確。将来の別作品がどのような仕様になるか分からず、何を共通化すべきか現時点では判断できない（当たり先のない一般化）
  - 現時点でChampions以外の具体的な対応要件がなく、YAGNI（You Aren't Gonna Need It）に反する
  - 抽象化レイヤーの導入・維持コストがChampions専用機能の開発速度を低下させる

### 選択肢3: 現状維持（暗黙のChampions専用、方針未定）

- 概要: 特に方針を定めず、現在の暗黙のChampions専用状態を維持する
- メリット: 何も変えないので即座のコストがかからない
- デメリット:
  - 「方針未定」そのものが設計負債になる。新機能追加やコードレビュー時に「これはChampions限定でよいのか」という判断が毎回発生する
  - 文書間の不整合（`ざっくり要求.md`の「SV」と`Claude要件定義.md`の「Champions」）が放置される

## 決定

[選択肢1] を採用する。

**PokemonToolsはPokemon Championsの仕様とデータを対象とする専用ツールとする。過去世代互換および将来作品を見越した汎用抽象化は初期スコープに含めない。**

理由:

1. 実装・要件定義・補助文書が既にChampions前提で揃っており、現状の追認である
2. 対戦環境の場はPokemon Championsに移りつつあり、本ツールの主要ターゲットである対戦補助機能はChampionsの仕様に完全対応すべきである
3. 将来の別作品の仕様は未知であり、今の時点で抽象化を試みても的外れになる可能性が高い
4. Champions専用とすることで、ドメインモデル・計算ロジック・UX全てにおいて最適な設計を追求できる

### 非目標

- 過去世代（SV、剣盾等）の計算式・育成システムへの互換性維持は行わない
- 世代差分を吸収する抽象化レイヤーは導入しない
- リポジトリ名・プロジェクト名の変更は本ADRの対象外とする（後述）

## 影響

### 正の影響

- スコープが明確化され、設計・レビュー・命名の判断基準が統一される
- 不要な抽象化を排除でき、開発速度が向上する
- Champions固有の仕様（能力ポイント制、テラスタル等）に最適化した設計を制約なく追求できる

### 負の影響

- 過去作品向けツールとしての流用性がなくなる
- 将来別作品に対応する場合、原則として新プロジェクトを立ち上げる必要がある。共通化の需要が明確になった時点で共通ライブラリ抽出を別ADRで検討する

### 必要な対応

- `ざっくり要求.md`の「対象世代: SV」をPokemon Championsに更新する
- 今後のドメインモデル・計算ロジック・マスターデータ設計はChampions仕様を正とする

### 非決定事項

- **プロジェクト名**: 「PokemonTools」のまま変更しない。リポジトリURL・CI設定・パッケージ名への影響が大きく、名称変更のコストに見合うメリットがないため。ただし、将来的に混乱が生じた場合は別ADRで再検討しうる

## 補足

- 本ADRはADR-0004の決定を前提とし、その延長線上にある
- Pokemon Championsの仕様詳細はADR-0004の補足欄（計算式・能力ポイントの制約）を参照
